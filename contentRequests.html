Content Requests

The Calls

This minute showing of requests is but the tip of the iceberg, however it does encompass the majority of the straight-up content requests. Keep in mind the table includes both requests to Wikipedia's RESTful API as well as it's legacy API.

# Request Format Description Example
1 GET /page/summary/{title} Get a text extract & thumb summary of a page https://en.wikipedia.org/api/rest_v1/page/summary/Koala
2 GET /page/related/{title} Get pages related to the given title https://en.wikipedia.org/api/rest_v1/page/related/Koala
3 GET /page/graph/png/{title}/{revision}/{graph_id} Get PNG graph images referenced in page revisions Requires graph ID
4 GET action=query&titles={title}&prop=revisions&rvprop=content&format=json Allows you to get information about a wiki and the data stored in it, such as the wikitext of a particular page https://en.wikipedia.org/w/api.php?action=query&titles=Sloth&prop=revisions&rvprop=content&format=json&callback=functionJSON
5 GET action=parse&format=json&prop=text§ion=0&page={title}&callback=functionJSON Parses content and returns parser output. https://en.wikipedia.org/w/api.php?action=parse&format=json&prop=text§ion=0&page=Koala&callback=functionJSON

Further Down The... No Rabbits. Just a Hole

It's obviously (really?) worth taking a look at Wikipedia's official (auto-generated) documentation, but there are three sections in particular that are (probably... No maybe)worth your while regarding content requests

What Where Why
action=query API:Query The action=query module is a fairly robust service (considering) with a reasonably indepth doc page.
action=parse API:Parsing The action=parse service apparently overlaps with another service or two, but is generally aligned with the content request objective and has significant (if not opaque) documentation.
Wikimedia REST API RESTBase RESTBase storage backends expose a RESTful storage API, similar to Amazon DynamoDB and Google DataStore, requesting content transformations from backend services when needed, handling high-volume static end points where most requests will be satisfied directly from storage.
