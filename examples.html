Holy Crap CORS!

Cross-origin resource sharing or "Shoot Me In The Foot CORS", as it is commonly known, is not specific to using the Wikipedia API, and actually is the recommended standard for a reason. It turns out CORS is for our insanity safety. In brief, the standard allows generally benign assets like fonts, images, videos, etc. to load from outside the origin of request (or cross-origin) (so we don't have to host our own YouTube or supply fonts), but what it doesn't allow is wonderful, powerful, potentially ruthless AJAX requests to eviscerate our local and network systems through maliciously implanted software. It's what we call a "win->lose" situation (but not quit "zero sum game")

Kidding aside, CORS affects how we must structure our requests as well as how we test them. We're going to tackle CORS with JSONP (the "P" is for "padding"!) and callbacks.

JSON vs. JSONP

I just want to be very clear on this, I'm like 200hrs into this whole 10k hr thing... I know virtually nothing about this. So disclaimers aside, this is how it works: When you make a JSONP request from your JavaScript file or nested within a script tag, the JSONP surreptitiously injects your DOM (typically in the head) with a script tag with the target URL. What it very specifically does NOT do is utilize the XMLHTTPRequest() method.

Let's build this query https://en.wikipedia.org/w/api.php?action=query&titles=Sloth&prop=images&format=json via a GET request (because we're not changing anything, just requesting something) using an ajax request (I will cover the Objects.keys()method below).

That's good and fine, but it doesn't actually provide the image URLs (womp womp)😞. You have to make an additional request that looks something like this:


https://en.wikipedia.org/w/api.php?action=query&titles=File:Bradypus.jpg&prop=imageinfo&iiprop=url&format=json

The next level is nesting/chaining the calls, but be sure to look into the best practice as it seems to be a somewhat contentious subject. Also, something something deferred objects.

Callbacks

JSONP responses are passed as an argument to a callback function. So you end up having to alter/append the target URL/GET request with &callback={functionName} or simply &callback=?. It may have been coincidental, but I found that I had to place the callback function parameter within the GET URL at the very end for Wikipedia to regularly accept it


The Callback Function = processJSON

https://en.wikipedia.org/w/api.php?action=parse&format=json&prop=text§ion=0&page=Sloth&callback=processJSON

If you're using a POST request (though you probably shouldn't be if you're just making a request for data... Serioulsy), jQuery subs in the success function for the ?.


To clarify, the purpose of all of this is to make a request to a site/service on a different server than that of the request-origin (which to be clear, is precisely what CORS is supposed to be preventing, so that "P" should be a "D" for "duplicitous"). Honestly though, I'm not sure why this is "okay", but it is definitely what is currently being done.

Accessing Known Properties Within a Dynamic JSON Object

Wikipedia's API, as I've mentioned, doesn't follow hard and fast conventions in the building of its pages, and so, when you request something, you can't bet on the responses to include for that which you are looking. More over, responses involve keys that you can't possibly know prior to reciept of the response, and hence, cannot define a specific route/path to a desired property within said object without a little help.


The Object.keys() method returns an array of a given object's own enumerable properties, in the same order as that provided by a for...in loop (the difference being that a for-in loop enumerates properties in the prototype chain as well).

It's worth checking out the full description, and even more useful to see this example on stack overflow, but the gist of it is, you can reference a property within a JSON object nested within an object parameter that you can't possibly explicitly reference because it's a unique identifier, it's a parameter that doesn't always show up for "these types of pages", or other shenanigans programmers think are okay to return as a response.
