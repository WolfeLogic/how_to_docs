Holy Crap CORS!

Cross-origin resource sharing or "Shoot Me In The Foot CORS", as it is commonly known, is actually the recommended standard for a reason. It turns out CORS is for our insanity safety. In brief, the standard allows generally benign assets like fonts, images, videos, etc. to load from outside the origin of request (or cross-origin) (so we don't have to host our own YouTube or supply fonts), but what it doesn't allow is wonderful, powerful, potentially ruthless AJAX requests to eviscerate our local and network systems through maliciously implanted software. It's what we call a "win->lose" situation (but not quit "zero sum game")

Kidding aside, CORS affects how we must structure our requests as well as how we test them. We're going to tackle CORS with JSONP (the "P" is for "padding"!) and callbacks.

JSON vs. JSONP

I just want to be very clear on this, I'm like 200hrs into this whole 10k hr thing... I know virtually nothing about this. So disclaimers aside, this is how it works: When you make a JSONP request from your JavaScript file or nested within a script tag, the JSONP surreptitiously injects your DOM (typically in the head) with a script tag with the target URL. What it very specifically does NOT do is utilize the XMLHTTPRequest() method. But the callback function is part and parcel. To clarify, the purpose is to make a request to a site/service on a different server than that of the request-origin (which to be clear, is precisely what CORS is supposed to be preventing, so that "P" should be a "D" for "duplicitous"). Honestly though, I'm not sure why this is "okay", but it is definitely what is currently being done.

Callback

JSONP responses are passed as an argument to a callback function. So you end up having to alter/append the target URL/GET request with &callback={functionName} or simply &callback=?. If you're using a POST request, jQuery subs in the success function for the ?.
